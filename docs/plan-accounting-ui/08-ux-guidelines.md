# Часть 8: UX-гайдлайны — чтобы бухгалтер и аналитик работали, а не боролись

Всё ниже — в рамках существующего дизайн-кита (чёрно-белый брутализм GRBPWR, underline-инпуты,
`Text`-варианты); никаких новых цветов/шрифтов/библиотек. Каждая практика привязана к
компоненту репо.

## 8.1 Персоны и их работа

| Персона | Частые задачи | Что критично |
|---|---|---|
| Бухгалтер | ручные проводки (banking-день), сверка, закрытие месяца, сторно | скорость ввода, невозможность тихой ошибки, понятные отказы бекенда |
| Финансовый аналитик | P&L/BS/TB по периодам, drill-down до первички, выгрузка в Excel | сравнимость периодов, копирование данных, доверие цифрам (final/preliminary) |
| Владелец | быстрый статус: прибыль, алерты, дельта сверки | вход из дашборд-алерта сразу в проблему |

## 8.2 Шесть принципов раздела

1. **No dead-end numbers.** Любая сумма кликабельна вниз до первички: TB/BS-строка → ledger →
   entry-деталка → заказ/партия/движение. Число, из которого некуда пойти, — баг UX.
2. **Период — это контекст, и он шарится.** Всё состояние периода/таба/счёта — в
   searchParams (уже так); заголовок раздела всегда показывает выбранный период. Ссылку из
   адресной строки можно кинуть в чат — собеседник увидит то же.
3. **Сигналы доверия всегда на виду.** На каждом отчёте: balanced-бейдж, статус периода
   (`OPEN`-джентл/`CLOSED`-замок — цифры финальны), caveats-блок. Аналитик должен с одного
   взгляда отличать предварительные цифры от финальных.
4. **Dr/Cr там, где думает бухгалтер; signed — там, где читает аналитик.** Двухколоночные
   debit/credit: журнал-деталка, TB, ledger. Однозначные суммы со знаком: P&L, BS, recon.
   Не смешивать в одной таблице.
5. **Ввод — keyboard-first.** Форма проводки полностью проходится с клавиатуры (Tab-порядок
   по строке, Enter в последнем поле = add line, Esc = закрыть с dirty-guard).
6. **Клиент не считает деньги.** Все суммы/итоги/балансы приходят с бекенда; клиентская
   арифметика — только live-футер формы (до сабмита) и подстановка баланса. Никаких
   пересчётов отчётов на клиенте.

## 8.3 Числовая типографика (правила `amount-cell.tsx` — единые для всех таблиц)

- `tabular-nums` (прецедент в репо: media-viewer/annotated-image) + `text-right` — цифры
  выравниваются в столбик, суммы сканируются вертикально.
- Всегда 2 знака: `1 234.50`, не `1234.5`. `formatBase`: `parseFloat(value).toLocaleString(
  'en-US', {minimumFractionDigits: 2, maximumFractionDigits: 2})` (прецедент toLocaleString —
  NewVsReturningPanel).
- Отрицательные — `-1 234.50` красным (`text-error`); скобочную нотацию не вводим (в
  остальной админке минус, консистентность важнее бухгалтерской традиции).
- `—` (em dash, muted) = «нет данных/строка не участвует»; `0.00` = настоящий ноль. Не путать.
- Итоговые строки: `border-t border-textColor` + `font-medium`; секционные подзаголовки —
  `Text variant='uppercase'` muted.
- Ширина колонок сумм фиксирована (`w-32`/`min-w-…`) — таблица не «дышит» при смене периода.

## 8.4 Форма ручной проводки — главный инструмент бухгалтера

- **Выбор счёта — `ComboField`** (`ui/form/fields/combo-field.tsx`: нативный datalist =
  подстрочный поиск бесплатно), options: строки `` `${code} — ${name}` `` по всем
  не-archived счетам. Сабмит извлекает ведущий 4-значный код; zod-валидация «код существует
  в списке» ловит свободный ввод. Бухгалтер набирает «sal» → видит `6330 — Salaries`.
- **Кнопка `balance` на последней строке**: подставляет |Σdebit − Σcredit| и
  противоположную сторону — типовой жест «добей проводку». Активна только при
  дисбалансе и base-режиме.
- **`duplicate line`** (иконка копии в строке) — вторая по частоте операция после add.
- Autofocus на description при открытии; Enter в последнем поле строки = add line;
  Esc = закрытие ЧЕРЕЗ dirty-guard: `form.formState.isDirty` → ConfirmationModal
  `Discard unsaved entry?`.
- Live-футер: `Σ debit · Σ credit · difference` — difference красным пока ≠ 0 (base-режим),
  сабмит-кнопка disabled при явном дисбалансе; при src-строках — muted-подпись
  `balance validated on server (FX)`.
- Пресеты (уже в 03) — рендерить строкой чипов-кнопок над формой с подписью `templates:`.
- После успеха — снэкбар `Entry created` и подсветка новой строки в списке (scroll-to +
  краткий bg-highlight — есть токен `--color-highlightColor`).

## 8.5 Отчёты — для аналитика

- **`copy table` на TB/P&L/BS/ledger** — killer-фича до появления CSV-экспорта (бэклог):
  кнопка собирает видимую таблицу в TSV-строку → `navigator.clipboard.writeText` (паттерн
  `ui/components/copyToClipboard.tsx`) → снэкбар `Table copied — paste into a spreadsheet`.
  Вставляется в Excel/Sheets с сохранением колонок. Дёшево (одна функция на данные, которые
  уже в памяти) и закрывает 80% потребности «выгрузить».
- **Sticky везде**: первая колонка (паттерн members-table) + `thead` (`sticky top-0
  bg-bgColor z-10`) — P&L на 13 месяцев скроллится в обе стороны без потери контекста.
- **Статус периода на отчёте**: рядом с датами бейдж `OPEN`/`CLOSED` (по `useAcctPeriods`
  для выбранного месяца; диапазон в несколько месяцев — `CLOSED` только если все закрыты).
- Пресеты периодов: `this month · last month · YTD` — единственные три; больше — шум.
- Drill-down: клик по строке счёта в TB/BS/P&L → ledger с этим счётом и тем же периодом
  (searchParams-переход, принцип №1).
- Recon: дельта ≈0 (< 0.01) — зелёный `matched`; иначе красная сумма + раскрытие items.
  Каждый item с `ref`-ссылкой (заказ/движение).

## 8.6 Periods — ритуал, а не форма

- **Pre-close светофор в confirm-модалке**: при открытии `close` сразу запросить
  `useReconciliation(месяц)` и показать три строки со статусами ДО попытки закрытия:
  `pending events: 0 ✓ · unposted movements: 2 ✗ · revenue delta: 0.00 ✓`. Кнопка
  `close period` активна всегда (бекенд — истина), но бухгалтер видит проблемы заранее,
  а не методом тыка. `notReady`-ответ после попытки рендерится в той же модалке.
- Тексты подтверждений — с последствиями (`Closed periods reject any new postings…`), уже в 05.

## 8.7 Микропаттерны

- Пустые состояния = «почему пусто + что сделать»: `no entries in this period — try “all
  time” or create a manual entry` (+кнопки). Никогда просто "no data".
- Загрузка: `Loader` + `min-h` контейнера таблицы — экран не прыгает.
- Ошибка запроса: короткий текст + кнопка `retry` (refetch) — не пустой экран.
- Снэкбары (sentence-case, консистентно): `Entry created / Entry reversed / Account created /
  Period closed / Period reopened / Table copied…` и ошибки `Failed to …` + текст бекенда.
- Tooltip'ы объяснений — только там, где термин неочевиден: `is_system` счёт, caveat-текст,
  `source` типа `opex_month`. Не тултипить очевидное.

## 8.8 Анти-паттерны (агентам: НЕ делать)

- Никаких графиков/спарклайнов в фазе 1 — аналитика живёт на главном дашборде.
- Никаких клиентских пересчётов отчётных сумм «для красоты» (принцип №6).
- Никакого localStorage для фильтров — searchParams уже решает и шарится.
- Никакой виртуализации таблиц (объёмы: 34 счёта, ≤50 строк на страницу).
- Никакого авто-поллинга — данные обновляет invalidation после мутаций + ручной refetch.
- Не изобретать компоненты, если есть в `src/ui` — сначала грепнуть кит.
